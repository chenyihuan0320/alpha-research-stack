# Codex 开发规则

本文件定义 Alpha Research Stack 的 Codex 协作规则。所有后续开发都应遵守这些约束。

## 基本流程

- 每次修改前必须先阅读 [docs/progress.md](docs/progress.md)，确认当前进度、已完成事项和下一步。
- 修改前先理解现有文件、目录结构和相关上下文，不凭记忆覆盖现有设计。
- 修改范围应紧贴用户目标，不做无关重构、不引入无关依赖、不实现未要求的业务代码。
- 每次修改后必须运行与变更相关的测试或检查；如果没有可运行的测试，应明确说明原因，并至少运行能够证明变更范围的检查命令。
- 修改完成后汇报变更文件、验证命令和未覆盖风险。

## 安全规则

- 不提交 API key、token、账号密码、Cookie、私钥、数据库连接串或任何真实凭证。
- 不把本地私有配置写入仓库。
- 需要配置凭证时，只能提交 `.env.example`、配置说明或占位变量。
- 不自动下单，不添加会直接触发真实交易的代码路径。
- 任何投资相关输出都必须保留“不构成投资建议”的边界。

## 开源复用优先

- 优先复用成熟开源项目，而不是从零写 demo。
- 新建模块前必须先判断是否已有可复用组件、库或参考项目。
- 对开源项目的引入需要记录用途、许可证、维护状态、适配成本、替代方案和风险。
- 项目差异化能力应集中在统一编排、研究工作流、信号治理、回测验证、复盘和报告层。

## LLM 使用边界

- LLM 可以用于研究摘要、行业梳理、反证搜索、财报解读、新闻归因、报告草稿和观点对照。
- LLM 不能凭感觉生成最终 confidence。
- 最终 confidence 必须来自可追溯的规则、数据特征、回测表现、样本外验证、风险校准和人工可审计的信号治理流程。
- LLM 输出必须能回溯到来源、数据、假设和不确定性，不能用无法验证的表述替代证据。

## 文档规则

- 架构、选型和流程变化应同步更新 `docs/` 下的相关文档。
- `docs/progress.md` 记录阶段进度、完成内容和下一步，不替代详细设计文档。
- 重要决策应写清楚取舍原因，避免只记录结论。
